# 7. Rework & Remake (การซ่อมแก้และการผลิตใหม่)

## ภาพรวม

เมื่อ NCR ถูกแก้ไข (Resolved) ด้วย Disposition เป็น **Rework** (ซ่อมแก้) หรือ **Remake** (ผลิตใหม่) ระบบจะดำเนินการแก้ไขบนใบสั่งงาน (Work Order) ที่เชื่อมโยงโดยอัตโนมัติ นี่คือสอง Disposition ที่สำคัญที่สุดที่ทีม QC ต้องเข้าใจ

---

## Rework (การซ่อมแก้)

### สิ่งที่ระบบทำ

Rework จะรีเซ็ตกระบวนการผลิตย้อนกลับไปยังขั้นตอนที่ไม่ผ่าน เพื่อให้ชิ้นงานสามารถผ่านกระบวนการใหม่ได้

### การดำเนินการอัตโนมัติ (เกิดขึ้นเมื่อ NCR → Resolved)

1. **รีเซ็ตขั้นตอนที่ไม่ผ่าน** เป็นสถานะ `pending` (ล้าง actual_start และ actual_end)
2. **รีเซ็ตขั้นตอนทั้งหมดหลังจากขั้นตอนที่ไม่ผ่าน** เป็น `pending`
3. **รีเซ็ต OPO ที่เชื่อมโยง** (Outside Processing Orders — ใบสั่งจ้างภายนอก) สำหรับขั้นตอนที่ได้รับผลกระทบกลับเป็น `pending` พร้อมล้างข้อมูลการส่งและรับ
4. **ตั้งค่า WO กลับเป็น `in_progress`** หากสถานะเป็น `completed` อยู่
5. **เพิ่มค่า `rework_count`** บน WO (เช่น จาก 0 เป็น 1 หรือ 1 เป็น 2)
6. **ตั้งค่า `source_ncr_id`** บน WO เพื่อเชื่อมโยงกับ NCR นี้
7. **เก็บ WO id** บน NCR เป็น `rework_work_order_id`

### สิ่งบ่งชี้ทางสายตา (Visual Indicators)

หลังจาก Rework, WO จะแสดง:
- ป้าย **Rework ×N** สีส้ม (โดย N คือจำนวนครั้งที่ซ่อมแก้)
- ป้ายจะปรากฏบน WO ในรายการ Work Orders, บอร์ด Kanban และบอร์ด Scheduling

[image: rework-badge — ภาพหน้าจอของการ์ด WO แสดงป้ายสีส้ม "Rework ×2"]

### ตัวอย่าง

```
NCR-2026-005 บน WO-2026-00042
  ข้อบกพร่องที่ขั้นตอนที่ 3 (CNC Machining) — มิติเกินค่าความคลาดเคลื่อน
  Disposition: Rework

  → ขั้นตอนที่ 3 ถูกรีเซ็ตเป็น pending
  → ขั้นตอนที่ 4 (Surface Treatment) ถูกรีเซ็ตเป็น pending
  → ขั้นตอนที่ 5 (Quality Check) ถูกรีเซ็ตเป็น pending
  → ขั้นตอนที่ 6 (Packing) ถูกรีเซ็ตเป็น pending
  → WO rework_count: 0 → 1
  → สถานะ WO ยังคงเป็น in_progress
  → พนักงานปฏิบัติการ (Operator) รันขั้นตอนที่ 3 ใหม่บนเครื่องจักร
```

## Remake (การผลิตใหม่)

### สิ่งที่ระบบทำ

Remake สร้างใบสั่งงานใหม่ (Child WO — ใบสั่งงานลูก) เพื่อผลิตชิ้นงานทดแทนตั้งแต่ต้น โดยปิด WO เดิม

### การดำเนินการอัตโนมัติ (เกิดขึ้นเมื่อ NCR → Resolved)

1. **ปิด WO เดิม** — ตั้งสถานะเป็น `completed` หากยังไม่อยู่ในสถานะสุดท้าย
2. **สร้างรายการธุรกรรมคลังแบบ Scrap** หาก WO มี `material_id` (บันทึกการสูญเสียวัสดุ)
3. **สร้างรหัส WO ใหม่** ผ่านระบบ Running Number
4. **สร้าง Child WO** ในสถานะ `released` โดยคัดลอกข้อมูลส่วนหัวทั้งหมด (ผลิตภัณฑ์, ลูกค้า, จำนวน, วัสดุ, ลำดับความสำคัญ, วันครบกำหนด)
5. **คัดลอกขั้นตอนเดิมทั้งหมด** เป็นขั้นตอนใหม่สถานะ `pending` บน Child WO
6. **สร้าง OPO อัตโนมัติ** สำหรับขั้นตอนจ้างภายนอกของ Child WO
7. **อัปเดต NCR** ด้วย `rework_work_order_id` = id ของ Child WO

### สิ่งบ่งชี้ทางสายตา (Visual Indicators)

Child WO จะแสดง:
- ป้าย **Remake** สีม่วง
- ป้ายจะปรากฏบน WO ในรายการ Work Orders, บอร์ด Kanban และบอร์ด Scheduling

[image: remake-badge — ภาพหน้าจอของการ์ด WO แสดงป้ายสีม่วง "Remake"]

### ตัวอย่าง

```
NCR-2026-008 บน WO-2026-00042
  ข้อบกพร่อง: การปนเปื้อนของวัสดุ — ทั้งล็อตใช้งานไม่ได้
  Disposition: Remake

  → WO-2026-00042 ถูกปิด (completed)
  → สร้างรายการธุรกรรม Scrap สำหรับวัสดุ Al 6061-T6
  → สร้าง WO-2026-00055 ใหม่ในสถานะ released
  → คัดลอกขั้นตอนทั้ง 6 เป็น pending
  → สร้าง OPO อัตโนมัติสำหรับขั้นตอนที่ 4 (Surface Treatment — จ้างภายนอก)
  → NCR เชื่อมโยงกับ WO-2026-00055
```

## Disposition อื่นๆ

| Disposition (การจัดการ) | การดำเนินการ | ใช้เมื่อใด |
|-------------|--------|-------------|
| **Scrap** (ทิ้ง) | ไม่มีการดำเนินการ WO อัตโนมัติ | ชิ้นงานใช้งานไม่ได้และต้องทิ้ง |
| **Use As-Is** (ใช้ตามสภาพ) | ไม่มีการดำเนินการ WO อัตโนมัติ | ข้อบกพร่องยอมรับได้ — ลูกค้ายินยอมหรือค่าเบี่ยงเบนอยู่ในเกณฑ์ |
| **Return to Vendor** (คืนผู้ขาย) | ไม่มีการดำเนินการ WO อัตโนมัติ | ข้อบกพร่องอยู่ในวัตถุดิบ — ส่งคืนซัพพลายเออร์ |

## การติดตาม Rework/Remake ในระบบ

### Quality Dashboard (แดชบอร์ดคุณภาพ)

- KPI **Rework Rate** (อัตราการซ่อมแก้) — เปอร์เซ็นต์ของ WO ที่เสร็จสิ้นแล้วที่ถูกซ่อมแก้
- KPI **Remake Rate** (อัตราการผลิตใหม่) — เปอร์เซ็นต์ของ WO ที่เสร็จสิ้นแล้วที่ถูกผลิตใหม่
- ทั้งสองคลิกได้ เชื่อมโยงไปยังรายการ WO ที่ถูกกรอง

### หน้า Work Orders (ใบสั่งงาน)

- ป้าย Rework และ Remake ปรากฏบนแถว WO
- กรองตาม Rework หรือ Remake เพื่อดู WO ที่ได้รับผลกระทบ

### หน้ารายละเอียด NCR

- การ์ด **Linked Work Order** แสดง WO ของ Rework/Remake พร้อมสถานะปัจจุบัน
- คลิก **View Work Order** เพื่อนำทางไปยัง WO โดยตรง

### Scheduling Board (บอร์ดการจัดตาราง)

- WO ที่ถูก Rework จะปรากฏอีกครั้งใน Unscheduled Sidebar (แถบด้านข้างงานที่ยังไม่จัดตาราง) เนื่องจากขั้นตอนที่ถูกรีเซ็ตต้องถูกจัดตารางใหม่
- Child WO จาก Remake จะปรากฏเป็นรายการใหม่ในแถบด้านข้าง

## แนวทางปฏิบัติที่ดี (Best Practices)

| แนวปฏิบัติ | เหตุผล |
|----------|-----|
| กำหนด Root Cause เสมอก่อนทำ Resolve | ช่วยป้องกันการเกิดซ้ำ; จำเป็นสำหรับ Rework/Remake |
| สร้าง NCR ทันทีหลังจากการตรวจสอบไม่ผ่าน | บริบทยังสด; ใช้กล่องโต้ตอบแจ้งเตือน NCR |
| ตรวจสอบ Linked WO หลังจาก Resolve | ตรวจสอบว่า Rework/Remake ดำเนินการถูกต้อง |
| ติดตามอัตรา Rework บน Dashboard | อัตรา Rework สูงบ่งบอกถึงปัญหาคุณภาพเชิงระบบ |
| ใช้ "Scrap" อย่างประหยัด | หมายถึงการสูญเสียวัสดุ — พิจารณาว่าสามารถ Rework ได้ก่อน |
