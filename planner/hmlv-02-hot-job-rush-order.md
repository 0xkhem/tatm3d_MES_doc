# การจัดการงานด่วน / Rush Order

Rush Order คือความเป็นจริงของโรงงาน HMLV ลูกค้าโทรมาตอนบ่ายสองต้องการชิ้นงานภายในวันศุกร์ หรือคำสั่งซื้อที่มีอยู่ถูกเร่งเพราะสายการประกอบของลูกค้ากำลังรอ คู่มือนี้ครอบคลุม Workflow ทั้งหมดสำหรับการจัดการงานด่วนโดยไม่ทำให้งานอื่นบนพื้นที่การผลิตพังทลาย

---

## ปัญหา Rush Order ใน HMLV

ต่างจากโรงงาน High-Volume ที่ตารางการผลิตถูกกำหนดล่วงหน้าหลายสัปดาห์ ผู้วางแผน HMLV ต้องจัดลำดับความสำคัญใหม่อยู่ตลอดเวลา ความท้าทายคือการทำเช่นนั้นโดยไม่:

- ทำให้ช่างประหลาดใจระหว่างทำงาน
- สร้าง Conflict บน Scheduling Board
- สูญเสียการมองเห็นว่างานใดถูกเลื่อนออกไป

TAM3D ให้คุณมีสามตัวควบคุม: **WO Priority**, **การจัดตารางใหม่ด้วยตนเอง** และ **การสื่อสารกับช่างผ่านระบบ Hold/Alert**

---

## Workflow: การจัดการ Rush Order

### ขั้นตอนที่ 1 — สร้างหรือระบุ Work Order

หากงานด่วนเป็นคำสั่งซื้อใหม่:
1. ไปที่ **Sales → Quotes** และสร้าง Quote หรือไปที่ **Sales → Sales Orders** โดยตรงและสร้าง SO
2. แปลง SO Line Item เป็น Work Order: เปิด SO → คลิก **Create Work Order** บน Line Item ที่เกี่ยวข้อง
3. WO ถูกสร้างในสถานะ `draft`

หากงานด่วนเป็น WO ที่มีอยู่แล้วที่ถูกเร่ง:
- ค้นหาใน **Production → Work Orders**

### ขั้นตอนที่ 2 — ตั้ง Priority เป็น High

เปิด Work Order → Edit

ตั้ง **Priority Level** เป็น `High` (หรือระดับสูงสุดที่โรงงานของคุณใช้) ซึ่งส่งผลต่อ:
- วิธีที่ WO ปรากฏใน Unscheduled Sidebar (เรียงตามความเร่งด่วน)
- พฤติกรรม Auto-Scheduling เมื่อใช้ Priority-Weighted EDD
- ลำดับ Task Queue ของช่าง (WO Priority สูงปรากฏก่อน)

บันทึก WO

### ขั้นตอนที่ 3 — Release Work Order

WO ต้องอยู่ในสถานะ `released` ก่อนจึงจะจัดตารางได้ หากยังอยู่ใน `draft`:
- เปิด WO → คลิก **Release**
- WO ต้องมีอย่างน้อยหนึ่งขั้นตอนจึงจะ Release ได้

### ขั้นตอนที่ 4 — จัดตารางด้วยตนเองบน Board

ไปที่ **Production → Scheduling Board**

ขั้นตอนของ WO ด่วนจะปรากฏใน **Unscheduled Sidebar** ทางซ้าย เรียงตามความเร่งด่วน WO Priority สูงปรากฏใกล้ด้านบน

**ตัวเลือก A — ลากและวาง:**
ลากการดำเนินงานจาก Sidebar ไปยังเครื่องเป้าหมายที่ช่วงเวลาที่ต้องการ Board แสดงความละเอียดระดับชั่วโมง (08:00–20:00) วางในช่องว่างแรกที่มี

**ตัวเลือก B — ใช้ Assign Dialog:**
คลิกการดำเนินงานใน Sidebar → คลิก **Assign** เลือกเครื่อง, วันที่, เวลาเริ่มต้น และระยะเวลา วิธีนี้แม่นยำกว่าสำหรับการจัดตารางที่ตึง

**การตรวจจับ Conflict:** หากคุณวางการดำเนินงานบนช่องที่ถูกครอบครองแล้ว Board จะไฮไลต์ Conflict ย้ายการดำเนินงานที่บล็อกออกเพื่อเปิดพื้นที่ (ดูขั้นตอนที่ 5)

### ขั้นตอนที่ 5 — ย้ายการดำเนินงานที่บล็อกหากจำเป็น

หากเครื่องเป้าหมายเต็มแล้ว คุณต้องเลื่อนบางอย่างออก ตัดสินใจว่างานที่มีอยู่ใดสามารถเลื่อนได้:

1. บน Timeline Grid คลิกการดำเนินงานที่ต้องการย้าย
2. ลากไปยังช่องหลัง หรือใช้ Popover → **Reassign** เพื่อย้ายไปยังเครื่องหรือวันที่ต่างกัน
3. ตรวจสอบ **QC Queue Panel** และ **Outside Processing Panel** เพื่อให้แน่ใจว่าการย้ายการดำเนินงานไม่สร้างปัญหา Downstream (เช่น การตรวจสอบ QC ที่จัดตารางไว้วันพฤหัสบดีตอนนี้ไม่มีอะไรให้ตรวจสอบ)

### ขั้นตอนที่ 6 — ใช้ Reschedule Mode สำหรับการหยุดชะงักขนาดใหญ่

หาก Rush Order ต้องการการ Re-optimize หลายเครื่อง:

1. คลิก ⚡ **Generate Schedule** บน Scheduling Board
2. เปิดใช้ **Reschedule Mode** — ซึ่งรวมการดำเนินงานที่จัดตารางแล้วในการ Optimization
3. ตั้ง `frozen_days` เพื่อปกป้องความมุ่งมั่นระยะใกล้ (เช่น 2 วัน = วันนี้และพรุ่งนี้ถูกล็อก)
4. เลือก **Priority-Weighted EDD** เป็น Dispatching Rule — เพื่อให้แน่ใจว่างาน Rush Priority สูงถูกจัดตารางก่อน
5. ตรวจสอบตารางที่เสนอใน Ghost Overlay ตรวจสอบแผง KPI สำหรับ Late Orders
6. หากยอมรับได้ คลิก **Accept** หากไม่ คลิก **Discard** และปรับด้วยตนเอง

### ขั้นตอนที่ 7 — สื่อสารกับช่าง

ช่างเห็น Task Queue ตามลำดับ Priority บน **Operator Panel** และ **Operator Home Dashboard** เมื่อ Rush WO ถูกจัดตารางและ Release แล้ว จะปรากฏที่ด้านบนของ Queue สำหรับช่างของเครื่องที่มอบหมาย

สำหรับการสื่อสารทันที:
- หากช่างกำลังทำงาน WO Priority ต่ำอยู่ พวกเขาสามารถ **Hold** WO ปัจจุบัน (Operator Panel → ปุ่ม Hold → เลือกเหตุผล "Rush Order / Expedite")
- WO ที่ Hold ปรากฏใน **Attention Panel** บน Home Dashboard เพื่อให้ผู้วางแผนติดตาม
- ช่างจึงหยิบงานด่วนจาก Task Queue ของตน

## Workflow: การเร่งคำสั่งซื้อที่กำลังดำเนินการ

บางครั้งงานด่วนไม่ใช่งานใหม่ — แต่เป็น WO ที่มีอยู่ที่ต้องการเสร็จเร็วขึ้น

### เพิ่ม Priority

แก้ไข WO → เพิ่ม Priority Level ซึ่งจะจัดเรียง Task Queue ของช่างใหม่ทันที

### ตรวจสอบว่าติดอยู่ที่ไหน

เปิดหน้ารายละเอียด WO ดูความคืบหน้าของขั้นตอน:
- ขั้นตอนรอเครื่องที่ยุ่งอยู่หรือไม่? จัดตารางใหม่ให้เร็วขึ้นหรือเปลี่ยนเครื่อง
- ขั้นตอนรอ Outside Processing กลับมาหรือไม่? ตรวจสอบ **Outside Processing Panel** — หากซัพล่าช้า ระบบจะแสดง OPO เป็น Overdue (แถบสีแดง)
- ขั้นตอนอยู่ใน Hold หรือไม่? ตรวจสอบเหตุผลการ Hold หากรอวัสดุหรือ Tooling ให้แก้ไขปัญหาและ Resume

### ใช้ Kanban สำหรับการ Triage ด้วยภาพ

**Production → Work Orders → Kanban** แสดง WO ทั้งหมดตามคอลัมน์สถานะ ลาก WO ที่เร่งไปด้านบนของคอลัมน์เพื่อส่งสัญญาณความเร่งด่วนด้วยภาพ (หมายเหตุ: นี่เป็นเครื่องมือช่วยมองเห็น ไม่ใช่การดำเนินการของระบบ — Priority ถูกตั้งบน WO เอง)

## Workflow: การสื่อสารความล่าช้ากับฝ่ายขาย

เมื่อ Rush Order เลื่อนงานอื่นออกไป ฝ่ายขายต้องรู้ว่าการส่งมอบใดมีความเสี่ยง

1. หลังจากจัดตารางใหม่ ไปที่ **Production → Dashboard → Schedule Adherence**
2. กรองตาม WO ที่ได้รับผลกระทบ รายงานแสดงวันที่เสร็จสิ้นที่วางแผนกับจริง
3. สำหรับ WO ใดก็ตามที่ตอนนี้จัดตารางให้เสร็จหลังวันครบกำหนด ให้เปิด Sales Order ที่เชื่อมโยง
4. **Delivery Promise Signal** บน SO จะอัปเดตเป็นสีเหลืองหรือแดงตามตารางใหม่
5. แชร์สิ่งนี้กับทีมขายเพื่อให้พวกเขาสื่อสารกับลูกค้าที่ได้รับผลกระทบ

## เคล็ดลับสำหรับสถานการณ์ Rush ที่เกิดซ้ำ

**สร้าง Process Route "Fast Track"** หากชิ้นงานบางประเภทถูกเร่งบ่อย ให้สร้าง Route ที่ง่ายขึ้น (`RT-PROTOTYPE-FAST`) ที่ข้ามขั้นตอนที่ไม่สำคัญ (เช่น CMM Inspection → Visual Check เท่านั้น) มอบหมาย Route นี้เมื่อสร้าง Rush WO

**เก็บ Buffer Slot บนเครื่องที่ดีที่สุดของคุณ** สำรอง 1–2 ชั่วโมงต่อวันบนเครื่องที่มีความสามารถมากที่สุดเป็น Capacity ที่ไม่ได้จัดตาราง เมื่อ Rush Job มาถึง คุณมีช่องว่างพร้อมโดยไม่ต้องเลื่อนอะไร

**ใช้ Delivery Promise Signal เชิงรุก** เมื่อฝ่ายขายสร้าง SO ใหม่ที่มีวันส่งมอบตึง Signal จะเปลี่ยนเป็นสีเหลืองหรือแดงทันทีหากโรงงานไม่สามารถทำได้ ซึ่งแสดง Conflict ก่อนที่จะกลายเป็นวิกฤต

**ตั้งค่า Abnormal Reason Category "Rush"** เมื่อช่าง Hold งานเพื่อทำ Rush Order พวกเขาสามารถเลือก "Rush Order / Expedite" เป็นเหตุผลการ Hold ซึ่งให้ข้อมูลว่า Rush Order รบกวนการผลิตปกติบ่อยแค่ไหน — มีประโยชน์สำหรับการสนทนาการวางแผน Capacity กับฝ่ายขาย
