# คู่มือกลยุทธ์การจัดตาราง

Scheduling Board คือเครื่องมือที่ทรงพลังที่สุดใน VulcornOne — และเป็นเครื่องมือที่ต้องตัดสินใจมากที่สุด คู่มือนี้อธิบายว่าเมื่อไหรควรจัดตารางด้วยตนเองกับอัตโนมัติ วิธีอ่านและดำเนินการกับบอร์ด 3 เลน และ Capacity Utilization Strip กำลังบอกอะไรคุณ

---

## Manual กับ Auto-Scheduling: กรอบการตัดสินใจ

คำตอบสั้นๆ: ใช้ Auto-Scheduling สำหรับงานส่วนใหญ่ ใช้ Manual สำหรับข้อยกเว้นและงานเร่งด่วน ทั้งสองแนวทางออกแบบมาให้ทำงานร่วมกัน ไม่ใช่แข่งขันกัน

### เมื่อไหรควรจัดตารางด้วยตนเอง

**Rush Order และงาน Priority สูง** เมื่องานต้องอยู่บนเครื่องเฉพาะในเวลาเฉพาะ ให้ลากไปวางเองเลย Auto-Scheduler Optimize สำหรับคิวทั้งหมด — มันไม่รู้ว่างานนี้ต้องเริ่ม 08:00 วันจันทร์เพราะลูกค้ามาตรวจสอบตอนเที่ยง

**หลังจากเครื่องจักรเสีย** เมื่อ EQP-0003 เสีย คุณต้องย้ายงานไป EQP-0004 ทันที นั่นคือการมอบหมายใหม่แบบเจาะจง ไม่ใช่การ Re-optimize ทั้งหมด เปิด Operation Bar ที่ได้รับผลกระทบ มอบหมายใหม่ทีละรายการ เสร็จ

**เมื่อคุณมีความรู้เกี่ยวกับพื้นที่การผลิตที่ระบบไม่มี** Scheduler รู้ชั่วโมงโดยประมาณและ Setup Group มันไม่รู้ว่าพี่โบ้เป็นคนเดียวที่รันโปรแกรม 5 แกนสำหรับ Bracket ไทเทเนียมได้ หรือ CMM กำลัง Calibrate เช้าวันอังคาร วางงานเหล่านั้นด้วยตนเองรอบข้อจำกัดที่คุณรู้

**โรงงานเครื่องเดียวหรือคิวเล็กมาก** ถ้ามีเครื่อง 3 เครื่องและงาน 8 งาน การวางด้วยตนเองใช้เวลา 5 นาทีและให้การควบคุมเต็มที่ Auto-Scheduling เพิ่มคุณค่าเมื่อความซับซ้อนเกินกว่าที่คุณจะจัดการในหัวได้ — ประมาณ 5+ เครื่องและ 20+ การดำเนินงานที่ยังไม่ได้จัดตาราง

### เมื่อไหรควรใช้ Auto-Scheduling

**เช้าวันจันทร์กับ Backlog เต็มสัปดาห์** หลังสุดสัปดาห์ คุณมี WO ที่ Release กองรอการกระจายตลอดสัปดาห์ Auto-Scheduling ด้วย Slack (Critical Ratio) จะวางอย่าง Optimal ในเวลาไม่ถึงนาที

**หลังจากการหยุดชะงักครั้งใหญ่** เครื่องเสียครึ่งวัน NCR หลายรายการ Reset WO หลายใบ Rush Order สับเปลี่ยน Priority — ตารางการผลิตตอนนี้ยุ่งเหยิง เปิด Reschedule Mode ตั้ง frozen_days เพื่อปกป้องความมุ่งมั่นวันนี้ รัน Engine มันจะ Re-optimize ทุกอย่างนอก Frozen Zone

**เมื่อต้องการเปรียบเทียบสถานการณ์** สร้างด้วย EDD บันทึก KPI สร้างด้วย Setup-Aware Grouping เปรียบเทียบ Ghost Overlay ให้คุณประเมินก่อน Commit ซึ่งเป็นไปไม่ได้ด้วยตนเอง

**เมื่อการ Optimize เวลา Setup สำคัญ** ถ้าโรงงานสูญเสีย 3+ ชั่วโมงต่อวันไปกับการเปลี่ยน Setup Setup-Aware Grouping สามารถกู้คืน Capacity ได้มาก Engine คำนวณสิ่งนี้ข้ามทุกเครื่องพร้อมกัน — คุณไม่สามารถทำซ้ำด้วยตนเองได้

### Hybrid Workflow (แนะนำ)

1. วางงานเร่งด่วน/Rush ด้วยตนเองก่อน
2. ตั้ง frozen_days = 1 หรือ 2 เพื่อปกป้องการวางเหล่านั้น
3. รัน Auto-Schedule พร้อม Reschedule Mode เปิดสำหรับส่วนที่เหลือ
4. ตรวจสอบแผง KPI — ดู Late Orders
5. Accept ถ้ายอมรับได้ หรือปรับ 1–2 งานที่มีปัญหาด้วยตนเองแล้ว Accept

วิธีนี้ให้การควบคุมในสิ่งที่สำคัญและประสิทธิภาพสำหรับทุกอย่างที่เหลือ

## การอ่านบอร์ด 3 เลน

บอร์ดซ้อนสามเลนที่ Align เวลาตามแนวตั้ง พวกมันแชร์มุมมองสัปดาห์เดียวกันเพื่อให้คุณเห็นการไหลทั้งหมดของงานข้ามทั้งสามเลนพร้อมกัน

```
┌─────────────────────────────────────────────────────────┐
│  เลน 1: Timeline Grid                                   │
│  การกลึงในโรงงาน — แถวเครื่องจักร × คอลัมน์ชั่วโมง      │
├─────────────────────────────────────────────────────────┤
│  เลน 2: Outside Processing Panel                        │
│  งานผู้รับเหมา — Gantt Bar ตามผู้รับเหมา                 │
├─────────────────────────────────────────────────────────┤
│  เลน 3: QC Queue Panel                                  │
│  คิวการตรวจสอบ — การ์ดแบ่งตามวัน                         │
└─────────────────────────────────────────────────────────┘
```

### เลน 1: Timeline Grid

นี่คือที่ที่คุณทำงานส่วนใหญ่ แต่ละแถวคือเครื่องจักร แต่ละคอลัมน์คือชั่วโมง (08:00–20:00) Operation Bar แสดงงานที่จัดตาราง

**การอ่าน Operation Bar:**
- ความกว้าง Bar = ระยะเวลา (กว้างกว่า = งานนานกว่า)
- สี Bar = WO Priority หรือสถานะ
- ป้ายกำกับ Bar = WO Code + ชื่อชิ้นงาน
- Hover = Popover พร้อมรายละเอียดทั้งหมด: WO, ขั้นตอน, ช่าง, ชั่วโมงที่วางแผน, ชั่วโมงจริงถ้าเริ่มแล้ว

**Overlay แผนกับจริง** เมื่อช่างเริ่มขั้นตอน Bar จะแสดง Overlay ชั่วโมงจริง ถ้าจริงรันนานกว่าที่วางแผน Bar จะแสดงตัวบ่งชี้ Overrun นี่คือสัญญาณ Real-time ว่างานล่าช้า

**ตัวบ่งชี้ Conflict** ถ้าสองการดำเนินงานทับซ้อนบนเครื่องเดียวกัน Bar ทั้งสองจะถูกไฮไลต์เป็นสีแดง แก้ไข Conflict โดยย้ายการดำเนินงานหนึ่งไปยัง Time Slot หรือเครื่องอื่น

**คำเตือน Dependency Violation** ถ้า Op20 ถูกจัดตารางให้เริ่มก่อน Op10 เสร็จ (บน WO เดียวกัน) บอร์ดจะแสดงตัวบ่งชี้คำเตือน ระบบจะไม่ป้องกันคุณจากการจัดตารางแบบนี้ แต่จะ Flag ปัญหาเชิงตรรกะ

### เลน 2: Outside Processing Panel

แสดง OPO ที่ Active ทั้งหมดเป็น Gantt Bar จัดกลุ่มตามผู้รับเหมา Bar ครอบคลุมตั้งแต่วันที่ส่งออกถึงวันที่คาดว่าจะกลับ

**รหัสสี:**
- เทา = Pending (ยังไม่ได้ส่ง)
- ส้ม = Shipped (ชิ้นงานอยู่กับผู้รับเหมา ภายในหน้าต่างการกลับที่คาดหวัง)
- แดง = Overdue (เกินวันที่คาดว่าจะกลับ ชิ้นงานยังไม่กลับมา)

**สิ่งที่ต้องทำกับ Bar สีแดง:**
1. คลิก Bar → ดูรายละเอียด OPO (ผู้รับเหมา, WO, วันที่คาดว่าจะกลับ)
2. ติดต่อผู้รับเหมา
3. อัปเดตวันที่คาดว่าจะกลับถ้าพวกเขาให้ ETA ใหม่
4. ตรวจสอบเลน 1 — มีขั้นตอนในโรงงานที่ถูกบล็อกอยู่หรือไม่? ถ้าใช่ Operation Bar ใน Lane 1 อาจต้องย้ายไปวันที่ช้ากว่า

**จำนวนใน Header** Header ของแผงแสดง "X active, Y late" ถ้า Y > 0 คุณมี OPO Overdue ซึ่งเป็นตัวเลขเดียวกับที่ปรากฏใน Attention Panel บน Home Dashboard

**การยุบแผง** ถ้าไม่มี Outside Processing ที่ Active ให้ยุบเลน 2 ด้วยปุ่ม Chevron เพื่อให้เลน 1 มีพื้นที่แนวตั้งมากขึ้น

### เลน 3: QC Queue Panel

แสดงขั้นตอน WO ที่กำลังเข้าใกล้หรืออยู่ที่ขั้นตอน QC/Inspection แบ่งตามวัน การ์ดปรากฏเมื่อขั้นตอน WO มีคำสำคัญ QC (QC, Quality, Inspection, CMM, Check) และวันที่มาถึงโดยประมาณอยู่ในมุมมองสัปดาห์

**การอ่านการ์ด QC:**
- การ์ด = WO หนึ่งรายการที่กำลังเข้าใกล้การตรวจสอบ
- คอลัมน์วัน = วันที่มาถึง QC โดยประมาณ (ตามการเสร็จสิ้นขั้นตอนก่อนหน้าหรือวันสิ้นสุดที่วางแผน)
- ตัวบ่งชี้สีแดง = วันครบกำหนด WO ผ่านไปแล้ว (Overdue)

**วิธีใช้:**
- ตรวจสอบคอลัมน์วันพฤหัสบดีเช้าวันจันทร์ — นั่นคือภาระงาน QC สำหรับวันพฤหัสบดี ถ้ามี 8 การ์ด เอิร์ธ (ผู้ตรวจสอบ QC) ต้องรู้
- ถ้าการ์ดแสดงสีแดง WO ล่าช้าแล้ว Escalate ทันที
- QC Queue Panel เป็น Read-only — คุณไม่สามารถลากการ์ดได้ มันเป็นเครื่องมือมองเห็น ไม่ใช่เครื่องมือจัดตาราง

### Cross-Lane Hover Linking

Hover บน WO ใดก็ได้ในเลนใดก็ได้ — WO เดียวกันจะถูกไฮไลต์ข้ามทั้งสามเลนพร้อมกัน (วงแหวนสีน้ำเงิน) และทุกอย่างอื่นจะหรี่ลงเป็น 35% Opacity นี่คือวิธีที่คุณติดตามคำสั่งซื้อเดียวตลอดการเดินทางทั้งหมดโดยไม่ต้องสลับมุมมอง

ตัวอย่าง: Hover บน Anodizing OPO Bar ใน Lane 2 → การดำเนินงานการกลึงใน Lane 1 และการ์ด CMM Inspection ใน Lane 3 ทั้งสองจะถูกไฮไลต์ คุณเห็นภาพรวมทั้งหมดของคำสั่งซื้อนั้นในมองเดียว

## Capacity Utilization Strip

ด้านล่างแต่ละแถวเครื่องจักรใน Lane 1 มีแถบสีบางๆ แสดง Capacity Utilization สำหรับแต่ละวัน นี่คือหนึ่งในสัญญาณที่ดำเนินการได้มากที่สุดบนบอร์ด

### วิธีอ่าน

แถบแสดงว่า Capacity รายวันที่มีของเครื่อง (08:00–20:00 = 12 ชั่วโมง) ถูกจัดตารางกี่เปอร์เซ็นต์

| สี | Utilization | ความหมาย |
|----|------------|---------|
| เขียว | 0–70% | มี Capacity — รับงานเพิ่มได้ |
| เหลือง | 70–90% | กำลังเต็ม — มีพื้นที่จำกัดสำหรับการเพิ่ม |
| แดง | 90–100% | ใกล้หรือถึง Capacity — ไม่มีพื้นที่สำหรับ Rush Job |
| แดงเข้ม | >100% | Overbooked — มี Conflict อยู่ แก้ไขทันที |

### วิธีดำเนินการ

**เครื่องสีเขียวข้างๆ เครื่องสีแดง** คุณมีความไม่สมดุลของโหลด เครื่องสีเขียวมี Capacity ว่างในขณะที่เครื่องสีแดง Overload ตรวจสอบว่าการดำเนินงานบนเครื่องสีแดงสามารถมอบหมายใหม่ให้เครื่องสีเขียวได้หรือไม่ (ต้องการ Equipment Type เดียวกัน) ลากไปวาง

**ทุกเครื่องสีแดงวันพุธ** วันพุธมีปัญหา Capacity ถ้า Rush Order ใหม่มาสำหรับการส่งมอบวันพุธ คุณต้องทำงานล่วงเวลาหรือเลื่อนบางอย่างออก Utilization Strip บอกคุณสิ่งนี้ก่อนที่คุณจะให้คำมั่นสัญญากับลูกค้า

**สีเขียวตลอดทั้งสัปดาห์** คุณมี Capacity สำรอง นี่อาจดี (Buffer สำหรับ Rush Order) หรือเป็นสัญญาณว่าโรงงานของคุณมีโหลดน้อยเกินไป ตรวจสอบ Delivery Promise Signal บน Quote ที่รอดำเนินการ — ถ้าเป็นสีเขียวทุกอย่าง คุณรับงานเพิ่มได้

**เครื่องหนึ่งสีแดงเสมอ เครื่องอื่นสีเขียวเสมอ** คุณมีเครื่องที่เป็น Bottleneck นี่คือข้อจำกัดของคุณ ทุกการตัดสินใจการจัดตารางควรให้ความสำคัญกับการป้อนงานให้เครื่องนั้น พิจารณา:
- การดำเนินงานใดสามารถย้ายไปยัง Equipment Type ต่างกันได้หรือไม่?
- Bottleneck เกิดจากเวลา Setup หรือไม่? ใช้ Setup-Aware Grouping เพื่อลดการเปลี่ยน Setup บนเครื่องนั้น
- เครื่องต้องการกะที่สองหรือไม่?

### Utilization กับแผง KPI

Utilization Strip แสดง Utilization ต่อเครื่องต่อวัน แผง KPI (หลังรัน Auto-Schedule) แสดง **Avg Utilization** ข้ามทุกเครื่องสำหรับ Horizon ทั้งหมด สิ่งเหล่านี้บอกเรื่องราวต่างกัน:

- Avg Utilization 78% กับเครื่องหนึ่งที่ 100% = คุณมี Bottleneck ที่ซ่อนอยู่ในค่าเฉลี่ย
- Avg Utilization 78% กับทุกเครื่องที่ 70–85% = โหลดที่สมดุลและดี

ตรวจสอบ Strip ควบคู่กับค่าเฉลี่ยแผง KPI เสมอ

## การเลือก View Mode ที่เหมาะสม

บอร์ดมีสอง View Mode: **By Machine** และ **By Line Item**

### By Machine (ค่าเริ่มต้น)

แสดงแถวเครื่องจักร ดีที่สุดสำหรับ:
- การตัดสินใจจัดตารางประจำวัน
- การระบุ Conflict และ Overload ของเครื่อง
- การติดตาม OPO และ QC Lane (มองเห็นได้เฉพาะใน Mode นี้)
- Drag-and-Drop Assignment

ใช้ Mode นี้สำหรับ 90% ของงาน

### By Line Item

แสดงแถว WO/Line Item แทนแถวเครื่อง แต่ละแถวคือ WO หนึ่งรายการ และ Bar แสดงขั้นตอนตลอดเวลา ดีที่สุดสำหรับ:
- การตรวจสอบ Timeline End-to-End ของคำสั่งซื้อเฉพาะ
- การตอบ "WO-2026-00042 จะเสร็จเมื่อไหร่?"
- การนำเสนอตารางการส่งมอบให้ฝ่ายขายหรือลูกค้า

OPO และ QC Panel ถูกซ่อนใน Mode นี้ — มันเป็นมุมมอง Order Timeline ล้วนๆ

## จังหวะการจัดตารางรายสัปดาห์

จังหวะรายสัปดาห์ที่สม่ำเสมอลดภาระทางความคิดของการจัดตาราง นี่คือรูปแบบที่ใช้ได้กับโรงงาน HMLV ส่วนใหญ่:

**เช้าวันจันทร์ (30 นาที):**
1. ตรวจสอบ Attention Panel — แก้ไข Hold และ OPO Overdue
2. กรอง WO ที่ครบกำหนดสัปดาห์นี้ — ระบุงานที่มีความเสี่ยง
3. วาง Rush Order ใหม่ด้วยตนเอง
4. รัน Auto-Schedule สำหรับสัปดาห์ (Slack Rule, frozen_days = 1)
5. Accept ตาราง

**เที่ยงวันพุธ (15 นาที):**
1. ตรวจสอบ Utilization Strip — เครื่องใดกำลังเป็นสีแดงสำหรับวันพฤหัส/ศุกร์?
2. ตรวจสอบ OPO Panel — มีการกลับที่ Overdue ที่จะบล็อกขั้นตอน Downstream หรือไม่?
3. ตรวจสอบ QC Queue — ภาระงานตรวจสอบวันพฤหัส/ศุกร์จัดการได้หรือไม่?
4. ปรับด้วยตนเองถ้าจำเป็น

**บ่ายวันศุกร์ (15 นาที):**
1. ตรวจสอบ Schedule Adherence Report — แผนกับจริงสัปดาห์นี้เปรียบเทียบอย่างไร?
2. บันทึก WO ที่รันเกินเวลาที่วางแผน — อัปเดต standard_time_minutes ถ้าเกินอย่างสม่ำเสมอ
3. ระบุ WO ที่ไม่เสร็จ — นำไปสู่ตารางสัปดาห์หน้า

จังหวะนี้รักษาตารางให้ทันสมัยโดยไม่ใช้เวลาทั้งวันของผู้วางแผน

## ข้อผิดพลาดการจัดตารางที่พบบ่อย

**จัดตารางทุกอย่างด้วยตนเอง** กับ WO ที่ Active 30+ รายการ การจัดตารางด้วยตนเองใช้เวลาหลายชั่วโมงและให้ผลลัพธ์ที่ไม่ Optimal ใช้ Auto-Scheduling สำหรับงานส่วนใหญ่ Manual สำหรับข้อยกเว้น

**รัน Auto-Schedule โดยไม่มี frozen_days** ถ้าคุณวางงานเร่งด่วนด้วยตนเองแล้วรัน Auto-Schedule โดยไม่มี frozen_days Engine อาจย้ายพวกมัน ตั้ง frozen_days ≥ 1 เสมอเมื่อมีการวางด้วยตนเองที่ต้องปกป้อง

**ละเลย Utilization Strip** การให้คำมั่นสัญญาวันส่งมอบโดยไม่ตรวจสอบว่าเครื่องเป้าหมายมี Capacity หรือไม่คือวิธีที่คุณสร้าง Late Order ตรวจสอบ Strip ก่อนยืนยัน Rush Order ใดๆ

**ไม่ยุบ Lane ที่ไม่ได้ใช้** ถ้าไม่มี Outside Processing สัปดาห์นี้ Lane 2 ใช้พื้นที่แนวตั้งโดยเปล่าประโยชน์ ยุบมัน เช่นเดียวกับ Lane 3 ถ้า QC Queue ว่าง

**Accept Auto-Schedule แรกโดยไม่เปรียบเทียบ** สร้างสองสถานการณ์ด้วย Rule ต่างกันและเปรียบเทียบ KPI ใช้เวลา 2 นาทีและมักเผยให้เห็นตัวเลือกที่ดีกว่าอย่างมีนัยสำคัญ
